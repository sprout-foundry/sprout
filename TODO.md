## SP-089: No valid way to enter a sudo password when a shell command needs it
_Spec: none yet (inline diagnostic from user report 2026-06-28)_

Bug / UX / Security (High, ~1 day): when an agent-issued shell command invokes `sudo` (or any other command that reads a password from the controlling TTY — `passwd`, `su`, `ssh-keygen` passphrase, `gpg` symmetric, `openssl passwd`, etc.), sprout has no supported way to surface a password prompt to the user. The shell tool either (a) hangs forever waiting on stdin that the agent never sends, (b) gets killed by an inactivity timeout and surfaces a confusing "command timed out" error, or (c) — depending on the tool classification — gets blocked by the security classifier as a `sudo` risk and never runs at all. None of those paths give the user a chance to actually type their password. The WebUI side has the same gap: terminal sessions over WebSocket can't surface the TTY read.

- [x] SP-089-1: Audit the three failure modes — (a) hang-on-stdin, (b) timeout, (c) classifier block — and decide which path the fix lives in. Likely two layers: (i) the security classifier must allow `sudo`/`passwd`/`su`/etc. when an interactive TTY is available AND a password-prompt handler is registered, (ii) the shell tool must detect the password-prompt pattern (`Password:` / `[sudo] password for` / `Enter passphrase`) on the child's stdout and route to a registered `PasswordPrompter` instead of waiting on the dead stdin pipe. _Audit complete (2026-06-28). Findings:_
  - _(c) classifier block is the dominant failure mode_ — `ClassifyToolCall("shell_command", {"command": "sudo apt update"})` returns `SecurityDangerous` with `RiskCategoryPrivileged`. The user now has a zombie background process to manually kill, with no UI indication of what to do._

Layer 2: shell tool's silent-mode path (which doesn't connect stdin) routes via a new `runShellCommandWithPasswordPrompter` that wraps the child, scans stdout for password-prompt patterns, and pipes the prompter's response to the child's stdin.
